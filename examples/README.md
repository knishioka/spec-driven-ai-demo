# デモシナリオ集

このディレクトリには、仕様書駆動開発デモ用のシナリオ例を用意しています。

## 使い方

各シナリオファイルを読んで、`/issue-draft` に入力する文言と期待される流れを確認してください。

## シナリオ一覧

| シナリオ | 所要時間 | 難易度 | 内容 |
|---------|---------|--------|------|
| [01-kebab-case.md](01-kebab-case.md) | 3-5分 | 簡単 | kebab-case変換関数の追加 |
| [02-truncate-text.md](02-truncate-text.md) | 5-7分 | 中程度 | テキスト切り詰め関数（省略記号付き） |
| [03-slugify.md](03-slugify.md) | 7-10分 | やや複雑 | URL-safe slug生成（Unicode対応） |

## シナリオの選び方

### 3-5分で終わらせたい場合
→ **01-kebab-case.md** を推奨

- シンプルな文字列変換
- エッジケースが少ない
- テストも直感的

### しっかり見せたい場合（5-7分）
→ **02-truncate-text.md** を推奨

- ビジネスロジックが明確（UI表示での文字数制限）
- 複数のAcceptance Criteriaが必要
- Non-goalsで範囲を明確に区切れる

### 完全な流れを見せたい場合（7-10分）
→ **03-slugify.md** を推奨

- Unicode処理など技術的な検討が入る
- 複数のエッジケース
- Plan段階で「やらないこと」が明確になる

## デモのコツ

1. **曖昧さを残す**: 最初の入力は意図的に曖昧にし、`/issue-draft` の質問機能を見せる
2. **AC確認**: spec.md生成後、Acceptance Criteriaが観測可能になっていることを強調
3. **スコープ確認**: plan.mdの「このPRでやること/やらないこと」を見せる
4. **検証強調**: PR_DRAFT.mdで実行コマンドと結果が記録されることを見せる

## カスタマイズ

デモ当日に別のアイデアが出た場合も、これらのシナリオをテンプレートとして使えます：

1. 同じ流れで別の関数名・要件に変更
2. `/issue-draft` の入力を微調整
3. 質問への回答を変えてみる

## 注意事項

- 各シナリオは独立しているため、順番に実施する必要はありません
- デモ前に一度通しで実行することを推奨します
- 生成されたファイル（spec.md, plan.md等）は `.gitignore` されているため、リセットは不要です
