# デモ本番ガイド

実際のデモでどのシナリオを選ぶか、どう見せるかの実践ガイド。

## シナリオ選択マトリクス

### 聴衆別の推奨シナリオ

| 聴衆 | 推奨シナリオ | 理由 |
|------|------------|------|
| **エンジニア（開発者）** | 03-slugify | 技術的判断（Unicode対応）の明示化に共感 |
| **PM/PdM** | 02-truncate-text | ビジネスコンテキスト→AC→Non-goalsの流れが明確 |
| **経営層/意思決定者** | 01-kebab-case | 短時間で全体像を見せ、ROIを示す |
| **混合（技術+ビジネス）** | 02-truncate-text | 両方の視点をカバー |

### 時間制約別の推奨

| 持ち時間 | シナリオ | 構成 |
|---------|---------|------|
| **3分** | 01-kebab-case | デモ2分 + 質疑1分 |
| **5分** | 02-truncate-text | デモ3.5分 + 質疑1.5分 |
| **10分** | 03-slugify | デモ7分 + 質疑3分 |
| **15分+** | 01 → 02（2回実演） | 簡単→複雑で理解深化 |

## シナリオ別デモ戦略

---

## 01: kebab-case（3-5分）

### 適したデモ状況

- 時間が限られている
- 初めて仕様書駆動開発を見る人が多い
- 「すぐできる」を示したい
- デモ後に別の議題がある

### デモの流れ（3分版）

```
0:00-0:30  導入「曖昧な要望から始めます」
0:30-1:30  /create-issue 実演（質問→回答→生成）
1:30-2:00  ISSUE_WITH_SPEC.md をざっと見せる
2:00-2:30  /resolve-issue 実演（plan.md確認→実装）
2:30-3:00  PR_DRAFT.md のVerificationを強調
```

### 強調ポイント

1. **曖昧→明確の変化**
   - 最初: "kebab-case変換関数が欲しい"
   - 結果: AC 3件 + Non-goals明確

2. **速さ**
   - 「3分で要望→実装→PR下書き完成」
   - 手作業なら30分かかる工程

3. **品質**
   - Verificationで実行結果を記録
   - テスト3件自動生成

### 予想される質問

**Q: これは本当に3分でできるの？**
→ A: はい。実際に今見せました。複雑な要件でも5-10分です。

**Q: Unicode対応は？**
→ A: このデモではスコープ外としました。Non-goalsに明記されています。必要なら別Issueで対応可能です。

**Q: エラーハンドリングは？**
→ A: シンプルなケースなので空文字返すだけ。ビジネス要件次第で追加できます。

## 02: truncate-text（5-7分）

### 適したデモ状況

- PM/PdMが多い
- ビジネスロジックの仕様化を見せたい
- 「なぜ作るか」「何を作らないか」を強調したい
- 中程度の複雑さを示したい

### デモの流れ（5分版）

```
0:00-0:30  導入「UI表示の実務課題から始めます」
0:30-2:00  /create-issue 実演
           - Q1: 背景（UI長文対応）
           - Q2: AC（切り詰め仕様）
           - Q3: Non-goals（単語境界は今回やらない）
2:00-2:30  ISSUE_WITH_SPEC.md レビュー
           - Backgroundの重要性
           - Non-goalsで将来のIssue明示
2:30-4:00  /resolve-issue 実演
           - plan.md でスコープ確認
           - エッジケース含むテスト6件
4:00-5:00  PR_DRAFT.md 確認
           - Verificationの透明性
           - Risk & Rollback
```

### 強調ポイント

1. **ビジネスコンテキストの明確化**
   - Why: "UI表示での長文対応"
   - 具体的なユースケース（カード、リスト）

2. **Non-goalsの価値**
   - 「単語境界での切り詰め」を明示的に除外
   - 将来的な拡張として記録
   - スコープクリープ防止

3. **制約の明示**
   - "省略記号は文字数に含める"
   - 仕様の曖昧さを排除

4. **エッジケースのカバレッジ**
   - 正常系4件 + 異常系2件
   - ValueError for negative length

### 予想される質問

**Q: 単語境界対応はいつやるの？**
→ A: Non-goalsに記録したので、Issue化して優先度付け可能です。今回は最小限の実装に集中しました。

**Q: HTMLタグが含まれる場合は？**
→ A: これもNon-goalsです。必要なら別関数`truncate_html()`として実装すべきです。

**Q: AC4件は多すぎない？**
→ A: エッジケースまで明確にすることで、実装者の判断ミスを防ぎます。観測可能な基準があるので、テストも書きやすいです。

## 03: slugify（7-10分）

### 適したデモ状況

- エンジニア中心の聴衆
- 技術的な判断プロセスを見せたい
- 「やらないこと」の判断を強調したい
- 完全な仕様書駆動開発を体験してほしい

### デモの流れ（7分版）

```
0:00-0:30  導入「ブログシステムでURL生成が必要」
0:30-2:30  /create-issue 実演
           - Q1: 背景（記事タイトル→URL slug）
           - Q2: Unicode（日本語どうする？）
           - Q3: 特殊文字（削除 vs 保持）
           - Q4: 制約（英数字のみ対応を明言）
2:30-3:30  ISSUE_WITH_SPEC.md レビュー
           - AC5件の詳細さ
           - Non-goals（Unicode音訳、最大長制限）
           - Constraintsで技術判断を記録
3:30-6:00  /resolve-issue 実演
           - plan.md で「やる/やらない」明確化
           - 正規表現処理の実装
           - テスト10件（エッジケース含む）
6:00-7:00  PR_DRAFT.md レビュー
           - Unicode非対応のリスク明示
           - Future Work として別Issue提案
```

### 強調ポイント

1. **技術的判断の透明化**
   - Unicode対応: 複雑すぎるので今回は見送り
   - 判断理由をConstraintsに記録

2. **複雑な要件の分解**
   - 5つのACで仕様を細分化
   - それぞれ独立してテスト可能

3. **将来の拡張性**
   - Non-goalsが次のIssue候補
   - 「今やらない」≠「永久にやらない」

4. **リスクの正直な開示**
   - PR_DRAFT.mdで「日本語タイトルは空文字」を明記
   - 隠さず、記録する

### 予想される質問

**Q: Unicode対応しないと使えなくない？**
→ A: その通りです。ただし最初のバージョンでは英語圏向けに限定し、シンプルに実装しました。Non-goalsに記録したので、次のIssueで`unidecode`ライブラリを使った拡張が可能です。

**Q: なぜ音訳ライブラリを最初から使わない？**
→ A: YAGNI（You Aren't Gonna Need It）原則です。まず動くものを作り、実際に必要になったら拡張します。仕様書駆動開発では「今必要なもの」に集中します。

**Q: AC5件は多すぎない？**
→ A: 複雑な仕様ほどACを細分化すべきです。「連続ハイフン統合」「先頭末尾削除」など、曖昧にすると実装者が迷います。

**Q: Future Workの別Issueは誰が作る？**
→ A: PMまたは実装者が優先度を見て起票します。Non-goalsに記録してあるので、Issue化が簡単です。

## デモ実施のコツ

### 事前準備

1. **リポジトリクリーン化**
   ```bash
   # 前回のデモファイルを削除
   rm -f ISSUE_*.md spec.md plan.md PR_DRAFT.md
   # テストが通ることを確認
   make test
   ```

2. **ターミナル設定**
   - フォントサイズ大きく（プロジェクター用）
   - カラースキーム: 白背景（視認性高い）
   - 履歴クリア（`clear`)

3. **ブラウザタブ準備**
   - GitHub Issues ページ
   - GitHub新規Issue作成ページ
   - README（参照用）

### 実演時のポイント

1. **最初の入力は意図的に曖昧に**
   - ❌ "snake_caseをkebab-caseに変換する関数で、空文字は空文字を返す"
   - ✅ "kebab-case変換関数が欲しい"
   → 質問機能を見せるチャンス

2. **質問への回答は自然に**
   - 台本通りではなく、その場で考えているように見せる
   - 「えーと、Unicode対応は...今回はいらないかな」

3. **ファイル生成時は内容を見せる**
   ```bash
   /create-issue
   # 生成されたら
   cat ISSUE_WITH_SPEC.md
   # または
   code ISSUE_WITH_SPEC.md  # VSCodeで開く
   ```

4. **plan.md確認は必ず声に出す**
   - 「このPRでやること...やらないこと...」
   - スコープゲートの重要性を強調

5. **Verification は実行コマンドを見せる**
   ```bash
   # PR_DRAFT.md に書かれたコマンドを実際に実行
   make test
   python -c "from demoapp.text_utils import to_kebab_case; print(to_kebab_case('hello_world'))"
   ```

### トラブル対応

**テストが失敗した場合**:
- 慌てない。「実際の開発でもこういうことありますよね」
- エラーメッセージを見せる
- 「/resolve-issueは失敗したら自動で修正してリトライします」

**質問で詰まった場合**:
- 「良い質問ですね。それはConstraintsに追加しましょう」
- その場でISSUE_WITH_SPEC.mdを編集して見せる

**時間が押した場合**:
- plan.md生成後、「ここから実装→テスト→PR生成まで自動です」
- 事前に用意したPR_DRAFT.mdを見せる

## デモ後のフォローアップ

### よくある質問への準備

**Q: 実際のプロダクトでも使えるの？**
→ A: はい。このデモはシンプルな例ですが、実際のプロダクトでも同じワークフローです。複雑な要件ほど仕様書駆動開発の価値が出ます。

**Q: チーム全員が使える？**
→ A: Claude Code Skillsを共有すれば、誰でも同じワークフローを使えます。属人化を防げます。

**Q: レビューはどうするの？**
→ A: PR_DRAFT.mdをベースにPR作成→通常のコードレビュー。Verificationセクションがあるので、レビュアーは実行結果を確認できます。

**Q: 既存プロジェクトに導入できる？**
→ A: 新規Issueから始められます。既存コードを一気に書き換える必要はありません。

```

```

**狙い**: 継続的な拡張、Non-goalsの活用
